Secure Software Development Lifecycle (SSDLC) Policy

Organization: ManageKube LLC
Application: Kubric Reach Collaboration IDE
Document Version: 1.0
Effective Date: February 28, 2026
Owner: Engineering & Security Team

1. Purpose

This document defines ManageKube's Secure Software Development Lifecycle (SSDLC) practices for the Kubric Reach application. It ensures security is embedded at every phase of development, from design through deployment and maintenance.

2. Scope

This policy applies to all code, infrastructure, and third-party integrations within the Kubric Reach platform, including the Zoom meeting integration.

3. SSDLC Phases

PhaseSecurity Activities
Requirements & DesignThreat modeling for new features; security requirements defined per integration (OAuth flows, token storage, RLS policies).
DevelopmentTypeScript strict mode enforced; ESLint security rules; secrets managed via environment variables (never hardcoded); all database queries parameterized via Supabase SDK.
Code ReviewAll changes require pull request review before merge; security-sensitive changes (auth, tokens, RLS) require senior review.
Build & DeployAutomated CI/CD via GitHub Actions; dependency vulnerability scanning; builds fail on critical vulnerabilities.
RuntimeRow Level Security (RLS) enforced on all database tables; JWT-based authentication via Supabase Auth; HTTPS enforced on all endpoints.
MonitoringEdge Function logging; database audit trails (crm_audit_log); error alerting via Resend email notifications.

4. Authentication & Authorization Controls

  • All users authenticate via Supabase Auth (email/password with JWT tokens).
  • OAuth 2.0 authorization code flow for Zoom integration with per-user token isolation.
  • Role-Based Access Control (RBAC) enforced at database level via RLS policies and helper functions (is_reach_member, get_reach_role).
  • Zoom tokens stored in reach_zoom_tokens table with RLS restricting access to token owner only.

5. Secret Management

  • All API keys and secrets stored in Supabase Edge Function environment variables.
  • No secrets committed to source code repositories.
  • Client-side code only has access to the Supabase anon (publishable) key.
  • GitHub tokens, Zoom credentials, and service role keys are server-side only.

6. Data Protection

  • All data in transit encrypted via TLS 1.2+.
  • Database hosted on Supabase (AWS infrastructure) with encryption at rest.
  • Zoom OAuth tokens stored in PostgreSQL with RLS — only the owning user can read/write their tokens.
  • No meeting recordings or participant PII are stored.

7. Dependency Management

  • Package dependencies managed via package.json with lock files for reproducible builds.
  • Regular dependency audits using npm audit.
  • Edge Functions use pinned Deno module versions from esm.sh and deno.land/std.

8. Incident Response

  • Security incidents reported to security@managekube.com.
  • Token compromise triggers immediate revocation via the zoom-oauth-callback?action=disconnect endpoint.
  • Stale/expired Zoom tokens are automatically purged on refresh failure.
